# Doc Quality Gate Integration

This project uses Doc Quality Gate (DQG) to review implementation documents against the codebase before coding begins.

## What DQG Does

DQG analyzes an implementation document (feature spec, implementation plan, architecture change, etc.) and cross-references it with the actual codebase to find:

- **Missing APIs** — Document references endpoints/routes that don't exist
- **Missing Models** — Document references data models/schemas not in the codebase
- **Missing Dependencies** — Required libraries not in package.json/requirements.txt
- **Conflicting Design** — Proposed changes conflict with current architecture
- **Redundant Work** — Document proposes building something that already exists
- **Architecture Mismatch** — Doesn't follow the project's established patterns
- **Quality Issues** — Contradictions, incomplete logic, missing requirements, edge cases

## How to Use

### Option 1: Slash Command (Recommended)

When you create an implementation document, run:

```
/dqg path/to/document.md
```

Or let DQG find the most recent document:

```
/dqg
```

### Option 2: Direct CLI

```bash
cd ~/Desktop/projects/framework/doc-quality-gate
source .venv/bin/activate
python -m app.cli review path/to/document.md --project /path/to/target/project
```

### Option 3: Wrapper Script

```bash
~/Desktop/projects/framework/doc-quality-gate/scripts/dqg-review.sh path/to/document.md
```

## Prerequisites

LiteLLM proxy must be running before using DQG:

```bash
cd ~/Desktop/projects/framework/doc-quality-gate
source .venv/bin/activate
litellm --config config/litellm/config.yaml --port 4000
```

## After Review

When `/dqg` returns results:

1. **Read the cross-reference issues first** — these show mismatches between the document and this codebase
2. **Fix the document** — Update the implementation document based on findings
3. **Re-run if needed** — `/dqg` again to verify fixes
4. **Implement** — Only start coding after the document passes the quality gate (score >= 8.0)

## Workflow Guidelines

- Always create an implementation document before starting significant coding work
- Use `/dqg` to validate the document against the codebase
- Address critical and high-severity issues before proceeding
- The revised document generated by DQG can be used as the implementation reference
- If DQG score is below 8.0, iterate on the document before implementing
